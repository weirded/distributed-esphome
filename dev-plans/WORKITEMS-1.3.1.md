# Work Items — 1.3.1 (hardening point release)

Theme: **Harden for 1.4.** A targeted hardening pass between 1.3 and 1.4. No new user features. Goal: close the highest-leverage regression blindspots, fragility hazards, and CLAUDE.md gaps surfaced by the post-1.3 audit, so 1.4's feature work (device create, firmware download, web serial, thread/IPv6, AI editor) lands on a foundation that catches breakage in CI instead of in production.

The audit's findings, in priority order:

1. **Server↔worker protocol is untyped dicts.** Magic-string keys crossing a network boundary. 1.4 will add new job fields; a typo will silently break deployed workers. Highest leverage.
2. **Scary code paths have no safety net.** `main.py` background loops, `version_manager.py` concurrency, OTA retry path, auth middleware edge cases, LIB.0 image-version branches, queue persistence corruption.
3. **Footguns waiting to fire.** Unlocked client globals, peer-IP auth trust, log-cap-after-append, silent options.json fallback, three CLAUDE.md violations already in the UI code.
4. **CLAUDE.md is mostly accurate but mixes enforceable invariants with aspirational guidance, has gaps the bug history proves we need, and is ~30% trimmable.**

## Workstream A — Typed contracts at the server↔worker seam

- [x] **A.1** *(1.3.1-dev.2)* **Define protocol message types** — `ha-addon/server/protocol.py` + byte-identical `ha-addon/client/protocol.py` (CI sync check via `tests/test_protocol.py::test_server_and_client_protocol_files_are_identical`). pydantic v2 `BaseModel` classes with `extra="ignore"` for forward compat. Covers `SystemInfo`, `RegisterRequest/Response`, `HeartbeatRequest/Response`, `DeregisterRequest`, `OkResponse`, `JobAssignment`, `JobResultSubmission`, `JobStatusUpdate`, `JobLogAppend`, `ProtocolError`. `pydantic>=2` added to both requirements files.
- [x] **A.2** *(1.3.1-dev.2)* **Add `protocol_version: int` field** — `PROTOCOL_VERSION = 1` in `protocol.py`, sent on `RegisterRequest` and `HeartbeatRequest` (default from the constant). Server-side `_check_protocol_version` helper in `api.py` returns a structured `ProtocolError(error="unsupported_protocol_version", ...)` with HTTP 400 on mismatch, instead of half-processing the payload.
- [x] **A.3** *(1.3.1-dev.2)* **Server-side validation on receive** — `api.py` parses every worker-facing endpoint body through the typed model via a shared `_parse_body()` helper. Malformed payloads return `ProtocolError(error="invalid_payload", reason="<field>: <msg>")` with HTTP 400. Rewired: register, heartbeat, deregister, `POST /jobs/{id}/result`, `POST /jobs/{id}/status`, `POST /jobs/{id}/log`. The `GET /jobs/next` response body is built from a `JobAssignment` model.
- [x] **A.4** *(1.3.1-dev.2)* **Client-side validation on receive** — `client.py` validates `JobAssignment` **before** extracting the bundle (polling loop in `worker_loop`), validates `RegisterResponse` and `HeartbeatResponse`, and builds every outgoing request from the typed pydantic model (`RegisterRequest`, `HeartbeatRequest`, `DeregisterRequest`, `JobResultSubmission`, `JobStatusUpdate`, `JobLogAppend`). `run_job` still takes a dict internally to avoid a larger refactor — the validation gate sits in front of it.
- [x] **A.5** *(1.3.1-dev.2)* **Round-trip + compatibility tests** — `tests/test_protocol.py` (29 tests): round-trip for every message type, forward-compat ("newer peer sent extra field — old peer must ignore"), backward-compat ("older peer omitted optional field — new peer must accept defaults"), rejection of malformed payloads (missing required fields, wrong types, unknown status literal), and the server↔client `protocol.py` byte-identity invariant.

## Workstream C — Fragility fixes

- [x] **C.1** *(1.3.1-dev.6)* **Lock client globals** — added `_state_lock` in `ha-addon/client/client.py` covering `_server_reachable` and `_auth_ok`. The four `_on_server_*` / `_on_auth_*` helpers now wrap their test-then-set in the lock so two threads racing on a connectivity transition can no longer log the same "went offline" line twice. `_active_jobs` was already locked; `_image_upgrade_logged` and `_update_attempts` are single-thread (heartbeat / main loop) and remain unlocked. Note: the workitem text mentioned `_selected_esphome_version` and the "component cache" — those don't exist in `client.py` (only in server `scanner.py`), so they were not in scope.
- [x] **C.2** *(1.3.1-dev.6)* **Harden `auth_middleware`** — added `_normalize_peer_ip()` in `main.py`: strips IPv6 zone IDs (`fe80::1%eth0`), unwraps IPv4-mapped IPv6 (`::ffff:172.30.32.2` → `172.30.32.2`), falls back to the raw string on parse failure. Both the peer IP and `HA_SUPERVISOR_IP` are now compared in canonicalized form. `peername=None` and `transport=None` paths handled with a try/except → fall through to token auth instead of crashing. Combined with the structured 401 logging from bug #3, C.2 is fully closed.
- [x] **C.3** *(1.3.1-dev.6)* **Cap log append BEFORE append** — the in-function cap in `JobQueue.append_log()` was already correct (it pre-computes a budget and only takes a slice, never concatenating the full input first). The remaining risk was at the request body parsing stage: a malicious or runaway worker could POST gigabytes which aiohttp would buffer before `append_log` ever ran. Added a `Content-Length` guard at the `append_job_log` handler in `api.py` that rejects anything over `4 × MAX_LOG_BYTES` (≈2 MB) with a structured `log_payload_too_large` error and HTTP 413.
- [x] **C.4** *(1.3.1-dev.6)* **`app_config.py` fail-loud** — `AppConfig.load()` now: (1) defines an explicit `_KNOWN_OPTION_KEYS` set and emits a WARNING for every unknown key in `options.json` (catches typos like `worker_ofline_threshold` immediately at startup); (2) logs ERROR with the offending value when a value is the wrong type (`int("garbage")` etc.) and falls back to the default; (3) logs INFO when `/data/options.json` doesn't exist so the "running outside HA add-on context" branch is visible; (4) emits a clear ERROR if the file parses as something other than a JSON object; (5) WARNs on the legacy `client_offline_threshold` alias; (6) WARNs when no token is configured anywhere (auto-generated path was previously silent). Every fallback path is now visible in the startup log.
- [x] **C.5** *(1.3.1-dev.6)* **Three CLAUDE.md violations in UI (pre-rewrite)**:

Future inputs go through these wrappers.
  - Replaced the inline `style={{ padding: 18 }}` in `ConnectWorkerModal.tsx` with the Tailwind utility `className="p-[18px]"`. Same value, no inline style escape.
- [x] **C.7** *(1.3.1-dev.6)* **Consolidate worker auth** — removed the duplicate `_check_auth` helper from `ha-addon/server/api.py` along with all 13 of its in-handler call sites. Every `/api/v1/*` request is gated by `auth_middleware` in `main.py` before it reaches a handler, so the in-handler check was dead code that risked drifting from the middleware (the middleware uses `constant_time_compare` plus the new structured 401 logging; the handler version did not).
