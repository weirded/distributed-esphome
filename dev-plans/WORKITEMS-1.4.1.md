## Rebrand: ESPHome Fleet

The product has grown beyond its original "distribute ESPHome compiles to faster machines" identity — 1.4 added fleet-scale management (version pinning, scheduled upgrades, device tags, Schedules tab). The current name "Distributed ESPHome" overweights the compile mechanism and underweights the actual user value. Rebrand to **"ESPHome Fleet"**.

**Ordering:** This section executes **before** the HA Native Integration section below. The integration's `DOMAIN` determines service names, entity IDs, events, and config entries — all of which are scripted against by users in automations. Changing the domain post-launch breaks every user's config. Doing the rebrand first lets us ship the integration with the `esphome_fleet` domain from day one.

**Scope: user-facing strings + integration domain.** Since HI.* ships in the same release with no users yet, we can also rename the in-progress integration domain (`distributed_esphome` → `esphome_fleet`) at zero migration cost. The rest stays the same — no breaking changes for existing add-on users.

**What stays the same:** repo name (`weirded/distributed-esphome`), Docker image names (`esphome-dist-server`, `esphome-dist-client`), add-on slug (`esphome_dist_server`), Python module names, logger names, and the YAML comment marker (`# distributed-esphome:` — parsed from users' existing YAML files; changing it would be a user-data migration).

## HA Native Integration

Custom integration that makes ESPHome Fleet a first-class HA citizen: native services callable from automations, `update` entities on the HA dashboard, and zero-config discovery via mDNS. Auto-installed by the add-on on startup (files copied to `/config/custom_components/`, user confirms via the Integrations UI). Integration domain is `esphome_fleet` (set in RB, above — all identifiers below use the new domain from day one).

**Auto-install mechanism:** The add-on already maps `homeassistant_config` (for reading ESPHome YAMLs). Change to `read_only: false` so we can write to `/config/custom_components/`. On startup, an s6 script compares the bundled integration version against what's installed and copies if newer. Then calls the Supervisor API to reload custom components. This is the same pattern used by other community add-ons — not an official API, but widely used and stable.

**Discovery:** The server advertises `_esphome-fleet._tcp` via mDNS. The integration's `manifest.json` declares a `zeroconf` matcher. HA shows "ESPHome Fleet discovered" → user clicks Configure → one confirmation screen → done. Falls back to manual URL entry if mDNS isn't working.
