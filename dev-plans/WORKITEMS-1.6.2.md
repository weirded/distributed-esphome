## IM — Install-path modernization (bug #82)

Bug #82 (1.6.1): Supervisor fails to install the add-on with `Image docker.io/library/docker:29.3.1-cli does not exist`. Root cause is upstream: Supervisor's `addons/build.py` picks the builder image as `docker:<HOST_DOCKER_VER>-cli` (`self.sys_docker.info.version`) and runs `docker buildx` inside it. The image DOES exist on Docker Hub (pushed 2026-04-03, actively pulled today); the pull failed silently (Supervisor's high-level `images.pull` swallows errors) and the generic `does not exist` message fires when the subsequent `containers.create` can't find it locally. Most probable triggers: Docker Hub anonymous rate limit (100/6h/IP) or transient Hub/egress hiccup — the pull wrapper surfaces both identically.

We're the only piece forcing Supervisor into the local-build path (`build.yaml` present + no `image:` key in `config.yaml`). Publishing prebuilt multi-arch add-on images and flipping `config.yaml` to `image:` moves every install to a single `docker pull` of our own GHCR image and eliminates the whole failure class. Side-effects that pay down existing debt:
- Kills the `build.yaml deprecated` Supervisor warning (cluster with TR.5, #8, SS.4).
- Unblocks real `@sha256:` digest pinning for the add-on's base image (Supervisor's `build_from` regex rejects digests silently; a GHCR image tag does not).
- Replaces 3–8 min of local `apt-get` + `pip install` + toolchain compile on the user's HAOS box with a seconds-long pull.
- Folds the TR.5 Debian-assertion guard into "belt-and-suspenders for the Supervisor-fallback-build path" rather than the primary defense.

This is hardening, not a feature — fits the 1.6.2 theme.

- [x] **IM.1** *(1.6.2-dev.6)* **`publish-addon.yml` — multi-arch add-on image to GHCR.** Landed. Matrix-per-arch workflow (`amd64` + `aarch64`, each pushed as a single-arch image so Supervisor's `{arch}` substitution resolves correctly), builds `ha-addon/Dockerfile` with `--build-arg BUILD_ARCH` matching Supervisor's production build semantics, pushes to `ghcr.io/weirded/<arch>-addon-esphome-dist-server:<VERSION>` plus `:latest` on `main` / `:develop` on `develop`. ## BD — Bundle discipline (cut over to `esphome bundle`)

Status quo before BD: `ha-addon/server/scanner.py:267–289` — `create_bundle()` did `base.rglob("*")` across `/config/esphome/` with filters only for macOS `._*` and `.DS_Store`. Every job claim shipped **the entire config directory** to the claiming worker — `.git/` (history + remote URLs + any wired-up push credentials), full `secrets.yaml` (every device's Wi-Fi, API keys, noise-PSKs — not just what the current target references), unreferenced YAMLs and packages, plus any `.esphome/` / `.pioenvs/` / `.pio/` caches if a user ever ran `esphome compile` in-place on the HA host.

ESPHome 2026.4+ ships `ConfigBundleCreator` (`esphome/bundle.py`, landed 2026-04-07) that does exactly the right thing: walks the validated config, ships only referenced files, filters `secrets.yaml` to the keys the bundled target actually references, emits a `manifest.json`. `.git/` and cross-device secrets don't ship by construction.

Scope-fit for 1.6.2: `.git/` + full `secrets.yaml` on every claim was a **latent secret-exfiltration vector** — a fleet operator adding a worker on a friend's Docker host effectively granted that friend read access to every device's secrets and the fleet's git history. Fits the "close shipped-but-broken paths" rule.

Approach: clean cut-over, no dispatcher, no opt-in, no fallback. Raise the ESPHome floor to 2026.4.0, call `ConfigBundleCreator`, delete the rglob path. Bundling is a server↔worker implementation detail the user never sees — the only user-visible surface is "ESPHome 2026.4+ required" in the release notes.

- [x] **BD.1** *(1.6.2-dev.15)* **Replace `scanner.create_bundle()` with `ConfigBundleCreator`.** Landed. `create_bundle(config_dir, target)` calls `_full_validate_config(target)` and passes the result to `esphome.bundle.ConfigBundleCreator`, returning `BundleResult.data`. Rglob loop + `._`/`.DS_Store` filters deleted. Caller in `api.py` updated to pass `job.target`, and the bundle call moved to `run_in_executor` so the heavy validate-then-bundle path doesn't block the event loop mid-claim. Worker side byte-identical on the wire. **Additional fix landing with BD.1:** `_full_validate_config` now calls `CORE.reset()` before every validation — ESPHome's `unique_ids` / `loaded_integrations` / `component_ids` are process-global and leak across successive validations on our 67-target fleet, causing false "Duplicate entity" errors on later targets. `esphome compile` CLI runs one target per process so never hits this; our server runs many sequentially and must mimic the fresh-process contract explicitly. Without the reset, BD.1 would have hit the no-fallback path for most targets on first boot.

- [x] **BD.2** *(1.6.2-dev.15)* **Raise ESPHome floor to 2026.4.0.** Landed. `constants.MIN_ESPHOME_VERSION = "2026.4.0"` is the authoritative floor; `scanner.ensure_esphome_installed` refuses any older version with an ERROR log + `_esphome_install_failed = True` so the UI banner surfaces the problem instead of hanging. `.github/workflows/compile-test.yml` bumped `ESPHOME_VERSION` to `2026.4.2` with an inline comment pointing at `MIN_ESPHOME_VERSION`. `ha-addon/DOCS.md` Getting Started section documents the floor and the refusal behavior. ## WL — Worker log visibility

The Workers tab shows worker state (online, parallel slots, current jobs) but no way to view the worker's own log output. Compile-job logs stream back and are viewable in Queue-History, but everything *outside* a compile job — startup errors, heartbeat failures, ESPHome install failures, OTA retries on a remote LAN, the "worker won't register" class of first-setup bug — is visible only on the host running the worker. For local workers (spawned inside the server add-on container) the output is merged into the add-on log alongside server noise; filtering to just the local worker's lines requires greppery. For remote workers on friend's-Docker-host / Raspberry-Pi / remote-office deployments, the operator has no visibility without SSH-ing to the worker's host — and by the time they're SSH-ing, they're already past the "is this thing working?" stage.

Scope-fit: rule (d) — shipping workers without a way to see why one's misbehaving is a first-setup operational break every first-time-remote-worker user hits. Kept in 1.6.2 (even though it's a new user-visible capability) because real users are currently stuck on "is this thing working?" with no recourse. Reuses the existing compile-log transport shape + xterm viewer so the new-capability surface is narrow.

Design approved 2026-04-23 — full spec at `dev-plans/designs/2026-04-23-worker-log-visibility.md`. Key constraint: **logs only traverse worker → server while a UI dialog is open.** Signalling rides on the existing heartbeat response (additive `stream_logs: Optional[bool]` — no `PROTOCOL_VERSION` bump per PY-6). Presence signal is per-worker WS subscriber count on the server; when count is 0, heartbeat response tells worker to stop pushing. When the dialog is open, worker pushes at 1 Hz to a dedicated endpoint that mirrors the compile-log transport line-for-line. The UI reuses `LogModal` parametrized over a tagged `source` union (`job` | `worker`) — one viewer, two callers.
