## IM — Install-path modernization (bug #82)

Bug #82 (1.6.1): Supervisor fails to install the add-on with `Image docker.io/library/docker:29.3.1-cli does not exist`. Root cause is upstream: Supervisor's `addons/build.py` picks the builder image as `docker:<HOST_DOCKER_VER>-cli` (`self.sys_docker.info.version`) and runs `docker buildx` inside it. The image DOES exist on Docker Hub (pushed 2026-04-03, actively pulled today); the pull failed silently (Supervisor's high-level `images.pull` swallows errors) and the generic `does not exist` message fires when the subsequent `containers.create` can't find it locally. Most probable triggers: Docker Hub anonymous rate limit (100/6h/IP) or transient Hub/egress hiccup — the pull wrapper surfaces both identically.

We're the only piece forcing Supervisor into the local-build path (`build.yaml` present + no `image:` key in `config.yaml`). Publishing prebuilt multi-arch add-on images and flipping `config.yaml` to `image:` moves every install to a single `docker pull` of our own GHCR image and eliminates the whole failure class. Side-effects that pay down existing debt:
- Kills the `build.yaml deprecated` Supervisor warning (cluster with TR.5, #8, SS.4).
- Unblocks real `@sha256:` digest pinning for the add-on's base image (Supervisor's `build_from` regex rejects digests silently; a GHCR image tag does not).
- Replaces 3–8 min of local `apt-get` + `pip install` + toolchain compile on the user's HAOS box with a seconds-long pull.
- Folds the TR.5 Debian-assertion guard into "belt-and-suspenders for the Supervisor-fallback-build path" rather than the primary defense.

This is hardening, not a feature — fits the Honest Gold theme.

Matrix-per-arch workflow (`amd64` + `aarch64`, each pushed as a single-arch image so Supervisor's `{arch}` substitution resolves correctly), builds `ha-addon/Dockerfile` with `--build-arg BUILD_ARCH` matching Supervisor's production build semantics, pushes to `ghcr.io/weirded/<arch>-addon-esphome-dist-server:<VERSION>` plus `:latest` on `main` / `:develop` on `develop`. ## BD — Bundle discipline (cut over to `esphome bundle`)

Status quo: `ha-addon/server/scanner.py:267–289` — `create_bundle()` does `base.rglob("*")` across `/config/esphome/` with filters only for macOS `._*` and `.DS_Store`. Every job claim ships **the entire config directory** to the claiming worker — `.git/` (history + remote URLs + any wired-up push credentials), full `secrets.yaml` (every device's Wi-Fi, API keys, noise-PSKs — not just what the current target references), unreferenced YAMLs and packages, plus any `.esphome/` / `.pioenvs/` / `.pio/` caches if a user ever ran `esphome compile` in-place on the HA host.

ESPHome 2026.4+ ships `ConfigBundleCreator` (`esphome/bundle.py`, landed 2026-04-07) that does exactly the right thing: walks the validated config, ships only referenced files, filters `secrets.yaml` to the keys the bundled target actually references, emits a `manifest.json`. `.git/` and cross-device secrets don't ship by construction.

Scope-fit for 1.6.2: `.git/` + full `secrets.yaml` on every claim is a **latent secret-exfiltration vector** — a fleet operator adding a worker on a friend's Docker host effectively granted that friend read access to every device's secrets and the fleet's git history. Fits Honest Gold's "close shipped-but-broken paths" rule.

Approach: clean cut-over, no dispatcher, no opt-in, no fallback. Raise the ESPHome floor to 2026.4.0, call `ConfigBundleCreator`, delete the rglob path. Bundling is a server↔worker implementation detail the user never sees — the only user-visible surface is "ESPHome 2026.4+ required" in the release notes.

- [ ] **BD.1 Replace `scanner.create_bundle()` with `ConfigBundleCreator`.** Load the validated config through ESPHome's own pipeline (reusing whatever entry point the #84 fix converges on — both workitems depend on running ESPHome's full validator on the server), instantiate `ConfigBundleCreator(validated_config)`, return `.data`. Delete the rglob loop and the `._` / `.DS_Store` filters (ESPHome's bundler wouldn't include them anyway). Worker side stays byte-identical: the bundle is still a tar.gz on the wire, `extract_bundle()` still tars into the per-slot build dir, and `esphome compile <target>.yaml` runs against the extracted tree as today — the bundle's `manifest.json` sits at the tree root and is inert from the compile-command perspective. No protocol changes, no worker changes, no settings, no UI.

- [ ] **BD.2 Raise ESPHome floor to 2026.4.0.** Bump `ESPHOME_VERSION` in `.github/workflows/compile-test.yml` from `2026.3.2` to the latest 2026.4.x patch. Document the floor in `ha-addon/DOCS.md` Installation ("ESPHome Fleet 1.6.2+ requires ESPHome 2026.4.0 or newer — older versions fail at job-claim time"). Add a version guard in `scanner.ensure_esphome_installed` that logs ERROR + sets `_esphome_install_failed = True` when the requested version parses to `< 2026.4.0`, so the UI banner reads "ESPHome version too old — upgrade to 2026.4+" instead of hanging on "installing…". Same guard prevents per-device `pin_version` downgrades below the floor.

- [ ] **BD.3 Regression tests.** `tests/test_scanner.py` — three cases pinning the security-relevant invariants:
  1. `test_bundle_excludes_git_dir` — seed tmp config dir with `device1.yaml` + `.git/config` carrying a fake push URL + `.git/objects/xx/abc…`; assert bundle contains zero `.git/*` entries.
  2. `test_bundle_filters_secrets_to_referenced_keys` — `device1.yaml` uses `!secret wifi_ssid` + `!secret wifi_password`; `secrets.yaml` carries those two keys plus `other_device_api_key: super_secret`; assert bundled `secrets.yaml` contains the two referenced keys and not the third.
  3. `test_bundle_omits_unrelated_targets` — config dir with `device1.yaml` + `device2.yaml`; bundle for `device1` does not ship `device2.yaml`.

- [ ] **BD.4 Docs.** `SECURITY.md` new "Closed in 1.6.2" entry — "Before 1.6.2, every job claim shipped the full `/config/esphome/` directory (including `.git/` and the complete `secrets.yaml`) to the claiming worker. Remote workers on third-party Docker hosts had read access to the fleet's full YAML history and every device's secrets. 1.6.2 switches to ESPHome's native bundle format, which ships only files the target references." `dev-plans/SECURITY_AUDIT.md`: entry under the data-exfiltration section. `ha-addon/CHANGELOG.md` under Bug fixes: user-language bullet naming the ESPHome floor bump alongside the security win ("Remote workers now receive only the files needed for the target being compiled, not the whole config directory. Requires ESPHome 2026.4+.").

## EH — ESPHome-delegation (stop hand-rolling what ESPHome already provides)

Audit on 2026-04-22 of `ha-addon/server/` + `ha-addon/client/` looking for places we re-implement functionality ESPHome ships as a library (`esphome.*`) or a CLI (`esphome <subcommand>`). Four findings below, ranked by (impact × ease-of-switch). Pairs philosophically with **#84** (same "stop hand-rolling; defer to ESPHome" shape for `domain:`-aware address resolution) and **BD.1** (delegate bundle creation to ESPHome) — every item here is latent-fragility hardening: code that works today but drifts the moment ESPHome renames a field, restructures an artifact, or ships a new target platform. Scope-fit: all four land under rule (b) ("fixes a shipped-but-broken path") in the sense that each is already *latently* broken for ESPHome releases we haven't seen yet. Known non-findings (audited + deliberately kept): `esphome run` as the single subprocess for compile+OTA — more robust than splitting into `esphome compile` + `esphome upload` (the consolidated path handles a whole class of edge cases the split would have to re-implement); the custom `_TagPreservingLoader` for round-tripping `!secret`/`!include` in the duplicate-device flow; the regex-substitution fallback used before the venv fires; `version_manager.py`'s per-venv pip install; and our HA entity-ID normalization / cron / integration-manifest code — none of those are ESPHome's domain.

- [ ] **EH.1 Use `esphome idedata` for firmware-artifact discovery.** `ha-addon/client/client.py:1225-1275` (`_collect_firmware_variants`) walks `.esphome/build/<name>/.pioenvs/<name>/firmware.{factory,}.bin` via hardcoded path templates. Stable for ESP32/ESP8266 today; breaks the moment ESPHome ships a target platform with a different PlatformIO layout (RP2040, nRF52, Zephyr, the new `host` platform). ESPHome's `esphome idedata <yaml>` (and `esphome.platformio_api.get_idedata()` Python API) emits JSON including `firmware_elf_path`, `firmware_bin_path`, and `extra_flash_images` (list of `{offset, path}` entries for bootloader/partition-table blobs). Fix: after a successful `esphome run`, invoke `<venv>/bin/esphome idedata <yaml>` and parse the JSON as the authoritative artifact manifest; retain today's path walk as a legacy fallback (probe once per job, cache the decision). Bonus: `extra_flash_images` unblocks a future USB-first-flash flow without further reinvention (currently we only archive `firmware.factory.bin` + `firmware.bin`; bootloader + partition table are silently dropped). Test: `tests/test_client_firmware_collection.py` across ESP32 (factory+ota), ESP8266 (ota only), and a synthetic RP2040 fixture; mocked `idedata` JSON drives the parse path and a real compile against `cyd-office-info` proves the wire.

- [ ] **EH.2 Prefer ESPHome's `storage.json` as source of truth for compiled-device metadata.** `ha-addon/server/scanner.py` (`_extract_metadata` ~848-912, `build_name_to_target_map` ~1060-1140, address resolution ~1011-1057) re-derives `esphome.name`, `friendly_name`, `platform`, `framework`, `board`, wifi/ethernet address, API encryption key, etc. from raw YAML every time. ESPHome writes all of this to `<build_path>/.esphome/storage.json` at the end of every successful compile (`esphome.storage_json.StorageJSON`) — HA's own ESPHome dashboard integration reads it as its source of truth. Fix: for any target that's been compiled at least once, `StorageJSON.load(config_path)` gets the authoritative post-validation values — including `last_ip_address` (ESPHome's own IP cache, cross-checkable against DevicePoller's cache) and `esphome_version` (which version was last used for that specific target, separate from the fleet-wide default). Raw-YAML parsing stays as the cold-start path for never-compiled targets and for UI views that must reflect uncommitted YAML edits before the next compile. Sequence this **after #84** — that work converges on running ESPHome's full validator on the server, which reshapes metadata extraction; EH.2 slots into the reshaped surface rather than being retrofitted into both. Test: unit coverage loading a real `storage.json` and asserting parity with the YAML-derived path; integration test confirming `storage.json` takes precedence when both exist.

- [ ] **EH.3 Replace magic-string config keys with `esphome.const` imports.** `ha-addon/server/scanner.py` and `ha-addon/client/client.py` reference `"esphome"`, `"name"`, `"wifi"`, `"ethernet"`, `"openthread"`, `"api"`, `"substitutions"`, `"packages"`, `"platform"`, `"framework"`, `"board"`, `"use_address"`, `"manual_ip"`, `"static_ip"`, `"domain"` as string literals scattered across many call sites. ESPHome's own code uses `CONF_ESPHOME`, `CONF_NAME`, `CONF_WIFI`, `CONF_ETHERNET`, `CONF_OPENTHREAD`, `CONF_API`, `CONF_SUBSTITUTIONS`, `CONF_PACKAGES`, `CONF_PLATFORM`, `CONF_FRAMEWORK`, `CONF_BOARD`, `CONF_USE_ADDRESS`, `CONF_MANUAL_IP`, `CONF_STATIC_IP`, `CONF_DOMAIN` from `esphome.const` — any upstream rename becomes an `ImportError` in our layer instead of a silent dict-miss that drops `friendly_name` / `use_address` / etc. for every user. Mechanical sweep; `ruff` + `mypy` + existing test suite catches typos. Scope the change to files where ESPHome is already on `sys.path` (i.e. anywhere touched after `_esphome_ready` fires) — the cold-start fallback paths keep the literal strings so they work before the venv is installed. Pairs naturally with EH.2 since `storage.json` reading will want the same constants.

- [ ] **EH.4 Simplify ESPHome version detection.** `ha-addon/server/scanner.py:207-228` (`_get_installed_esphome_version`) shells to `<venv>/bin/esphome version` and string-parses `"Version: X.Y.Z"` on stdout as the primary path; `importlib.metadata.version("esphome")` at lines 232-234 is a fallback. Once the venv's `site-packages` is on `sys.path` (which happens before `_esphome_ready` fires) the two paths return the same answer — the subprocess is redundant and ~50ms of fork+exec on a hot path. Fix: either (a) reorder so `importlib.metadata` is primary and subprocess is the disambiguator when we have reason to believe server-process Python is pointing at a different ESPHome than the venv, or (b) import `esphome.const.__version__` directly (single attr read, no subprocess, no parsing). Keep the current memoization. Lowest priority of the four; bundle with any nearby scanner.py work rather than as a standalone PR. Test: `tests/test_scanner_version.py` with and without the venv activated, asserting the two paths agree and the subprocess is skipped on the warm path.

## SI — Standalone independence (make Home Assistant optional)

ESPHome Fleet ships as an HA add-on (primary path) and as a standalone Docker image (supported — the repo ships `docker-compose.yml`; #83 closed the standalone Web UI auth UX). The standalone path works today but carries latent coupling to HA runtime: Supervisor-injected env vars, the HA entity poller, Ingress-header assumptions, `/data/options.json` loaders, the HA Core WebSocket client, and the custom-integration installer all expect "HA is present." Some sites already fall back cleanly; others silently no-op or produce confusing errors with no signal that HA-absence is the cause. Goal: every HA-coupled site is either (a) cleanly disabled when HA-absent with a clear WARNING, or (b) honestly documented as HA-required.

Scope-fit: rule (b). `docker-compose.yml` + #83 commit us to a working standalone path — SI is the hardening that makes that commitment real. Pairs with TP (truth-in-claims): a `docker-compose.yml` that ships broken edges is a claim the code doesn't fully back.

## WL — Worker log visibility

The Workers tab shows worker state (online, parallel slots, current jobs) but no way to view the worker's own log output. Compile-job logs stream back and are viewable in Queue-History, but everything *outside* a compile job — startup errors, heartbeat failures, ESPHome install failures, OTA retries on a remote LAN, the "worker won't register" class of first-setup bug — is visible only on the host running the worker. For local workers (spawned inside the server add-on container) the output is merged into the add-on log alongside server noise; filtering to just the local worker's lines requires greppery. For remote workers on friend's-Docker-host / Raspberry-Pi / remote-office deployments, the operator has no visibility without SSH-ing to the worker's host — and by the time they're SSH-ing, they're already past the "is this thing working?" stage.

Scope-fit: rule (b). Shipping workers without a way to see why one's misbehaving is a latent operational break — every first-time-remote-worker user hits it the first time registration fails and has nowhere to go. Reuses the existing compile-log transport shape + xterm viewer so the new-capability surface is narrow.

- [ ] **WL.1 Worker-side log-tail ring buffer.** `ha-addon/client/client.py` attaches a `logging.Handler` subclass that tees every log line (same stream that goes to stdout today) into a bounded `collections.deque` (`maxlen=2000`, ~2 MB ceiling) plus a monotonic line counter. Buffer is process-lifetime only — no disk persistence (remote workers stay stateless; no new writable volume requirement). Thread-safe: the handler may be called from any thread the logging framework dispatches from, so guard the deque with a `threading.Lock`. The buffer is the source the WL.2 push reads from.

- [ ] **WL.2 Heartbeat-piggyback log push.** Extend the heartbeat request model in `ha-addon/server/protocol.py` (plus the byte-identical client copy — PY-6) with an optional `log_tail: list[LogLine] | None` field carrying every line since the last successful heartbeat, capped at 500 lines per heartbeat (older lines stay in the client ring buffer until the next heartbeat drains them; steady-state producers can't outpace heartbeat cadence). Additive + backward-compatible — no `PROTOCOL_VERSION` bump. Server-side registry grows a per-worker ring buffer (2000 lines, TTL'd out 1h after last heartbeat) fed by the incoming heartbeats; `/ui/api/workers/<id>/logs?since=<cursor>` reads from it. No new auth-gated endpoint on the worker side; no second protocol surface. Local worker (spawned inside the server add-on) goes through the same heartbeat path as remote workers — uniform treatment, same UI, no special case.

- [ ] **WL.3 UI log viewer + coverage.** `ha-addon/ui/src/components/WorkersTab.tsx`: each worker row gets a "View logs" entry in its dropdown menu — lift the menu's `open` state out of the cell per the bugs-#2/#71 pattern (1 Hz SWR poll would otherwise slam it shut mid-click). Opens a shadcn `Dialog` containing the same xterm viewer the compile-log dialog uses (reuse, don't fork). While open, polls `/ui/api/workers/<id>/logs?since=<cursor>` at 2 Hz with a monotonic-cursor pattern so the tail advances without re-rendering the whole buffer; "Download logs" exports the full buffer as a timestamped `.log` file. Coverage: `tests/test_worker_log_buffer.py` (overflow drops oldest, monotonic counter survives retrieve, handler doesn't double-log to stdout, thread-safe concurrent append + drain), `tests/test_worker_log_protocol.py` (heartbeat round-trip with + without the field, oversized-payload truncation, server-side per-worker TTL eviction), `ha-addon/ui/e2e/worker-logs-dialog.spec.ts` (mocked API returns a fake log stream; dialog opens, tail cursor advances, Download produces a file with the expected content-disposition header).

- [ ] **#84** — `wifi.domain` / `ethernet.domain` ignored when resolving OTA target; worker pushes to `<name>.local` instead of `<name><domain>`. Filed 2026-04-22 (bakerkj, 1.6.1). ESPHome's wifi component injects `use_address = CORE.name + config[CONF_DOMAIN]` during *validation* (`esphome/components/wifi/__init__.py:342`, default `.local` at `:376`), so the canonical resolved string lives in `config["wifi"]["use_address"]` once validation completes. Our `ha-addon/server/scanner.py::_resolve_esphome_config` stops one step short — it runs `load_yaml` + `do_packages_pass` + `do_substitution_pass` but never triggers the wifi/ethernet validators, so `use_address` is never injected. `get_device_address` then hand-rolls a three-tier waterfall (`use_address` → `manual_ip.static_ip` → `f"{device_name}.local"`) that quietly drops `domain`. Symptom on bakerkj's box: target with `wifi: {domain: .my-domain.com}` → server hands the worker `device-name.local` → OTA fails at `Error resolving IP address of ['device-name.local']`. **Same waterfall is the single source of truth for the dashboard IP column.** `scanner.build_name_to_target_map` calls `get_device_address` once per target and hands the result to `DevicePoller` as `address_overrides` + `address_sources` (`device_poller.py:566–591`); every `Device` row's `ip_address` + `address_source` is seeded from there (`device_poller.py:614–624`), served via `/ui/api/devices`, and rendered by `DevicesTab.tsx` + `useDeviceColumns.tsx` (the "Address source: wifi_use_address" tooltip comes from the same field). The mDNS discovery loop can later override `ip_address` with an observed IP — but for users on non-`.local` domains, mDNS discovery won't fire at all, so the dashboard stays stuck on the wrong `.local` guess until the integration poll or the worker's OTA attempt exposes the mismatch. Fixing `get_device_address` fixes the OTA failure AND the dashboard IP column AND the integration device poller's poll target in a single place. Same function, same call graph, one fix.

  Fix shape: **stop hand-rolling; defer to ESPHome.** Extend `_resolve_esphome_config` to run ESPHome's full validator (`esphome.config.validate_config` or the closest public entry on the active version — probe once on module load, cache the reference), cache the validated config alongside mtime exactly as today, and rewrite `get_device_address` to read `config[<block>]["use_address"]` directly for whichever of wifi/ethernet/openthread is present (openthread uses a different knob — honor whatever ESPHome's validator puts in its equivalent field). The three source labels collapse to `{wifi,ethernet,openthread}_validated`, fully subsuming today's six wifi/ethernet/openthread_{use_address,static_ip} + mdns_default.

  Why ESPHome-native, not patch the waterfall: (a) `domain` is one missed field — DHCP hostname interactions with `fast_connect:`, thread address resolution, and any future ESPHome addressing knob would stay broken by the same mechanism the hand-rolled waterfall already demonstrates. (b) HT.3 already adopts "ESPHome-as-oracle" for the test side; using the oracle at runtime too closes the loop. (c) #20 (ESPHome 2026.4.0 `do_substitution_pass` reshape), #22 (friendly_name dropping on unresolved subs), and the BD.1 work all point the same direction — "we re-implemented an ESPHome pipeline step and it drifted." One more drift is cheaper to fix here than across N call sites later.

  Edge cases + fallback strategy: full validation is stricter than our current substitution-only pass — a typo'd component elsewhere in the file (e.g. `sensor: platform: dht112`) that resolves-but-doesn't-validate today still yields metadata via the partial resolver and would fail hard under full validation. Keep the hand-rolled waterfall as a **targeted fallback**: on `validate_config` failure for a target, log WARNING with the validation error, and return the legacy three-tier answer (honoring `use_address` + `manual_ip.static_ip`, falling back to `{name}.local`). A user with `domain:` set AND a typo elsewhere gets `.local` until they fix the typo — strictly no worse than today.

  Cost: validation is more expensive than substitution-only. Profile on the hass-4 fleet sample; if >50ms per-target after cache miss, introduce a coarser cache key (mtime + ESPHome version) rather than reverting the approach. Hot paths affected: device poller's `build_name_to_target_map`, bundle claim (§BD), UI targets list — all already tolerate `None` from `_resolve_esphome_config` today.

  Coverage: (1) HT.3 is the parametrized ESPHome-as-oracle suite — rename target `tests/test_ota_address_resolution.py` and fold in `wifi_domain.yaml`, `ethernet_domain.yaml`, `wifi_domain_via_substitution.yaml`, `wifi_domain_via_secret.yaml`. Oracle is `esphome.core.CORE.address` after validation, so every ESPHome change to domain handling auto-propagates. (2) `tests/test_scanner.py::test_get_device_address_falls_back_on_validation_error` — target YAML with a typo'd component, assert we get the waterfall fallback + WARNING log (belt-and-suspenders for the degradation path). (3) HT.4 sibling `e2e-hass-4/wifi-domain-ota.spec.ts`: target with `wifi.domain: .invalid-tld.test` — compile succeeds, job record's `ota_address` ends in `.invalid-tld.test`, OTA fails at the resolve step (intentional; assertion is on the resolved address, not a successful upload).

  Scope-fit: shipped-broken path — compiles produce firmware, then fail OTA on every user with a non-default `domain:`. In-scope for 1.6.2 under rule (b). TR-adjacent; no new workitem prefix needed. Pairs philosophically with BD.1 (defer to upstream ESPHome for config-file-set discovery) — both are "stop re-implementing ESPHome pipeline steps."
