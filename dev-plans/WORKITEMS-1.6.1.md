# Work Items — 1.6.1

Theme: **Home Assistant gold-standard polish.** Lift the add-on toward the highest practical Supervisor security score (AppArmor profile, declared `signed`, digest-pinned bases) and the bundled integration from `bronze` to `gold` on the HA Integration Quality Scale (diagnostics, repairs, system health, reauth/reconfigure flows, ≥95% test coverage). Plus the surface polish reviewers notice first — proper icon/logo sizes, multi-language translations, service-action docs, repo hygiene files.

Background: there is no single "Gold" tier for add-ons in the HA ecosystem — the operational definition for this release is **add-on security stars (max practical) + integration Quality Scale Gold + presentation/repo polish**. Today the security posture is already strong (cosign signatures, SBOM attestations, hash-pinned deps, full `SECURITY.md` threat model); the gap is in declarative surfaces (AppArmor, `signed: true`) and the integration's Gold-tier rules (currently `quality_scale: bronze`).

Some current `config.yaml` flags (`host_network: true`, `hassio_api: true`, `homeassistant_api: true`, `auth_api: true`) are deliberate trade-offs — not gaps to close. Each gets a one-paragraph justification in `DOCS.md` so the lower star count has a written rationale users can read.

## Security score (Supervisor stars)

- [x] **SS.1 AppArmor profile** *(1.6.1-dev.5, first-pass permissive)* — `ha-addon/apparmor.txt` lands + `apparmor: true` declared in `config.yaml`. Supervisor loads the named profile (`esphome_dist_server`, renamed by Supervisor to `local_esphome_dist_server` on install) so the add-on runs under AppArmor confinement rather than fully unconfined, which flips the Supervisor "AppArmor" security star. Dmesg confirms the profile is applied (`operation="profile_load"`).

  **First-pass is deliberately permissive** (`file,`, `capability,`, `network,`, `signal,`, `dbus,`, `unix,`, `mount,`, `pivot_root,`, `ptrace,`). Two earlier attempts with path-level allows hit denials — the stock `abstractions/python` include didn't cover `/usr/local/lib/libpython3.11.so.1.0` on our slim base (boot-time failure, `DENIED open`), and a broader `file,` with explicit `deny /config/...` carveouts passed boot but broke PlatformIO's `penv` bootstrap during compile. The working set of narrow rules varies by Python minor version + PlatformIO release + ESPHome toolchain, so itemising them would churn on every upstream bump. Permissive-but-attached buys the security-star credit today; tightening is tracked as future work against observed denial telemetry.

  Hass-4 smoke 17/17 green with the profile active. No DENIED entries in dmesg after profile replace.
- ~~**SS.2 `signed: true`**~~ WONTFIX *(1.6.1-dev.1)* — there is no `signed` field in Supervisor's add-on `config.yaml` schema. The only sibling concept is `codenotary:` (pre-2024 CAS signer integration), which is effectively abandoned and doesn't validate against Sigstore's cosign signatures. Our images *are* cosign-signed via keyless OIDC (`publish-server.yml` / `publish-client.yml`) and the signatures are verifiable externally with `cosign verify` — the `DOCS.md` → "Verifying what you're running" section has the command. - [x] **SS.4 Base image digest pinning** *(1.6.1-dev.5, partial — worker fully pinned, server blocked by Supervisor regex)*:
  - **Worker** (`ha-addon/client/Dockerfile`): digest-pinned since CF.3 in 1.6; digest re-verified this cycle against `docker buildx imagetools inspect python:3.11-slim`, unchanged from last release (`sha256:233de06753d30d120b1a3ce359d8d3be8bda78524cd8f520c99883bfe33964cf`).
  - **Server `ARG BUILD_FROM` default** (`ha-addon/Dockerfile`): digest-pinned this cycle. Local `docker build` without Supervisor now uses the pinned digest by default. In production Supervisor overrides this via `--build-arg BUILD_FROM=<value from build.yaml>`, so the ARG default only matters for local / CI reproducibility.
  - **Server `build.yaml`**: attempted the pin on current Supervisor (hass-4, 2026-04) — Supervisor silently rejects `@sha256:…` in the `build_from` map (the regex at `supervisor/validate.py` is unchanged from the CF.3-era observation). When the pin is present, Supervisor falls back to its default arch-base image (`ghcr.io/home-assistant/<arch>-base`, Alpine/musl) without surfacing a validation error; the add-on build then fails on our `apt-get` step (Alpine has no apt), which is how we detect the rejection. Kept build.yaml tag-only; the constraint has been re-documented in the file so the next attempt doesn't rediscover it. Full digest pinning at the Supervisor level remains **BLOCKED on upstream Supervisor** accepting `@sha256:…` in the `build_from` regex.
