# Work Items — 1.6.0

Theme: **Power-user fleet management.** Git-backed config version history, per-device compile history, archived-config recovery, key/value device tagging, disk budgeting, and declarative worker routing — the controls you need to run a growing fleet without losing track of what it's doing.

## Settings

Introduce an in-app Settings surface separate from Home Assistant's Supervisor-managed `options.json`. Reasons:

- Feature toggles (auto-commit on save, job history retention, disk budgets, future proxy config) don't belong in the Supervisor Configuration tab — they're product settings, not deployment plumbing. Editing via Supervisor also forces a full add-on restart per change, which is hostile for day-to-day toggles.
- Over time most user-facing configuration should live here rather than in the Supervisor options form. `options.json` keeps only the fields Supervisor genuinely needs (token, port, `require_ha_auth`).

**Storage model.** `/data/settings.json`, owned by the server. Atomic write (tempfile + `os.replace`). Loaded once into an in-memory singleton; every mutation rewrites the file and updates the singleton under a single asyncio lock.

**Bootstrap.** On first boot after the 1.6 upgrade, if `/data/settings.json` doesn't exist, seed it once from the current `options.json` values for fields that have moved (`job_history_retention_days`, `firmware_cache_max_gb`, `job_log_retention_days`). After that the file is the only source of truth — editing the moved field in Supervisor has no effect. Documented in `DOCS.md`.

**Immediacy.** Every setting change takes effect immediately. Server-side consumers read `get_settings().xxx` at decision time (not at startup), so the next scheduled retention run, the next git auto-commit, the next cache prune all see the new value on the next tick — no add-on restart. UI consumers subscribe via an SWR hook on `/ui/api/settings`; the drawer's `mutate()` after a PATCH propagates changes to other tabs on the next poll.

## Auto-Versioning (local git, zero config)

Every save creates a git commit automatically. Users get per-file history, diff, and rollback without touching git or configuring anything. The config directory becomes a local git repo on first startup; existing git repos are left intact.

**Why git:** Diff is free (`git diff`), history is free (`git log --follow`), rollback is `git checkout <hash> -- <file>` + a new commit, Monaco already has a diff editor in the bundle, and the `git` binary ships in the HA add-on base image (zero image size cost). The alternative (copy-on-write snapshots + DIY diff) is more code and worse tooling.

- [x] **AV.1 Auto-init** *(1.6.0-dev.4, refined dev.5)* — on server startup, if `/config/esphome/` is not a git repo, run `git init` + `git add -A` + initial commit "Initial commit by distributed-esphome". If it's already a git repo (user set it up themselves), skip init entirely — including the `.gitignore`, which is left untouched so Pat-with-git's curated file isn't mutated. Fresh-init still writes `.gitignore` with `secrets.yaml` + `.esphome/` safety-net entries. Pre-existing repo with no resolvable `user.name`/`user.email` (nothing in repo/global/system config) gets a one-time Fleet identity fallback so auto-commits don't emit git's "configured automatically" warning.
- [x] **AV.2 Auto-commit on save** *(1.6.0-dev.4, refined dev.5, completed dev.10)* — editor save + **every other user-initiated write path** (pin / unpin, meta, schedule set / delete / toggle / once, rename, delete, archive restore) now triggers `git add --all -- <file>` + `git commit -m "<action>: <file>"` in a debounced async task. Rename commits both old and new paths so the history reads as a delete-of-old + add-of-new. Action label is specific per endpoint (e.g. `pin: bedroom.yaml`, `schedule: bedroom.yaml`) so `git log --author="HA User"` reads like an activity feed. **Gated on `settings.auto_commit_on_save`** (live-read at commit time). **Author respects the repo's own `user.name`/`user.email`**. Debounce window of 2s coalesces rapid actions. `.archive/` added to fresh-init gitignore so soft-deletions don't double-track. Non-blocking: each endpoint returns immediately, commits happen async.
- [x] **AV.3 History endpoint** *(1.6.0-dev.10)* — `GET /ui/api/files/{filename}/history` returns `[{hash, short_hash, date, author_name, author_email, message, lines_added, lines_removed}]` via `git log --follow --numstat`. The `--follow` flag tracks renames. Paginated via `limit` (default 50, max 500) + `offset`. Empty list (not an error) if the file has no commit history yet. Parser uses `\x1f` (unit separator) between fields — `\x1e` (record separator) breaks `str.splitlines()` in Python which treats it as a line boundary; the fix is tested.
- [x] **AV.4 Diff endpoint** *(1.6.0-dev.10)* — `GET /ui/api/files/{filename}/diff?from=<hash>&to=<hash>` returns `{"diff": "<unified diff>"}`. Works for any combination of hashes: both → direct diff; only `from` → diff against HEAD; only `to` → diff working tree against `to`; neither → diff working tree against HEAD. Hash values validated as 4–40 hex chars so injected shell metachars can't leak into `git diff` (tested).

**Build order (remaining).** AV.6 is the gateway that makes rollback and manual-commit actually usable — the endpoints for both (AV.5, AV.11) are built *with* the history panel so the Restore button and the "Commit these changes" affordance are live the moment the UI ships. AV.7 follows as its own pass. AV.8 folded into AV.6 — the panel IS the diff viewer. AV.9 stays deferred until the file-tree editor exists.

- [x] **AV.5 Rollback endpoint** *(1.6.0-dev.11)* — `POST /ui/api/files/{filename}/rollback` body `{hash}`. Runs `git checkout <hash> -- <filename>` to restore the working-tree file to that commit's content. **If `settings.auto_commit_on_save` is true**, follows up with `git add + git commit -m "revert: <file> to <short-hash>"` so the rollback is itself a recorded commit. When auto-commit is false, the working tree is left dirty — the user is expected to review + commit via AV.11. Returns `{content, committed, hash, short_hash}`. Invalidates the scanner config cache. Validates `hash` as 4-40 hex chars. Tests: 4 unit + 2 integration covering both toggle states, malformed hashes, and non-repo safety.

- [x] **AV.6 History + diff panel** *(1.6.0-dev.11, absorbs the old AV.8)* — the full-featured per-file history UX, opened from the Editor modal's toolbar ("History" button) and from the Devices-tab hamburger menu ("View history…"). One drawer/sheet scoped to one file:

  1. **Header row.** File name, commit count, and an auto-commit status chip (on/off + link to the Settings drawer). If there are uncommitted changes to this file, a yellow banner above the content reads *"You have uncommitted changes. [Commit…]"* and clicking the button opens AV.11's flow.

  2. **Compare selector.** A pill bar near the top: `From: [—] ↔ To: [Current]`. Both pills are dropdowns listing every commit plus "Current" (=working tree). Changing either side re-renders the diff below. Defaults: `From` = HEAD's parent (or the oldest available commit), `To` = Current — so opening the panel shows "what changed since the last committed state."

  3. **Diff viewer.** Monaco's built-in `DiffEditor` (already in the bundle, just not imported — this is what the retired AV.8 was going to do). If the two sides are identical the editor shows a "no differences" empty state.

Shift/cmd-click a second row to set `From` to that commit instead. Per-row actions on hover/menu:
     - **Set as From / Set as To** — explicit compare
     - **Restore this version** — calls AV.5 with a confirmation dialog; shows what rollback will do (one-line summary: "Restore to {short_hash} from {relative time}. This will create a new `revert:` commit." or "…this will replace your uncommitted changes." when auto-commit is off).

Does NOT open inside the Editor modal (avoids a modal-in-modal situation); it's a separate drawer that can be opened with the Editor still underneath.

## Job History

Today the queue (`/data/queue.json`) IS the user's record of what the system did. Bug #18 removed time-based auto-pruning so history survives restarts, but per-target coalescing in `enqueue()` still clears *earlier* terminal jobs for a target whenever a new compile is enqueued (deliberate: one row per device on the Queue tab, not an ever-growing failure log). Net effect: users can only ever see the *latest* compile attempt per target, never the sequence of past attempts. Once they explicitly Clear the queue, even that disappears.

Add a persistent append-only job history that survives both coalescing AND explicit clears, powering: per-device "when did this last succeed?", fleet-wide success-rate rollups, compile-time trend lines, and audit trails of who-compiled-what-when. Separate from the live Queue (UI-wise and storage-wise), read-only from the user's perspective.

**Storage model:** SQLite (`/data/job-history.db`) rather than a growing JSON file. Per-job writes are cheap, indexed queries by target / timestamp / state are fast, and the file stays self-compacting. Add-on base image already has `sqlite3` as a stdlib builtin.

**Write model:** every time a job transitions to a terminal state (SUCCESS, FAILED, TIMED_OUT, CANCELLED), copy the finalized `Job` into the history DB. Coalescing still clears the queue entry but the history record stays. Explicit Clear on the Queue tab also leaves history intact.

- [x] **JH.1 Schema + DAO** *(1.6.0-dev.23)* — new `ha-addon/server/job_history.py`. SQLite schema `jobs(id TEXT PK, target, state, triggered_by, trigger_detail, download_only, validate_only, pinned_client_id, esphome_version, assigned_client_id, assigned_hostname, submitted_at, started_at, finished_at, duration_seconds, ota_result, config_hash, retry_count, log_excerpt)` with indexes on `(target, finished_at DESC)` and `(finished_at DESC)`. `log_excerpt` is the last ~2 KB (trimmed at a newline boundary). DAO is synchronous SQLite via a short-lived connection per call (sub-ms writes; no executor needed) with WAL mode on for read-while-write concurrency. Exposes `init()`, `record_terminal(job)` (upsert semantics — the OTA-patch path in `submit_result` needs to overwrite the initial row), `query(target, state, since, limit, offset)`, `stats(target, window_days)`, `last_per_target(targets)`, and `evict_older_than(days)`.

## Worker Update Documentation

After the SC.4 revert (bug #58 in 1.4.1), workers update manually: user notices the stale badge in the Workers tab and runs `docker pull && docker restart` on the worker host. Two gaps today:

1. **`ha-addon/DOCS.md`** (`line 21`) mentions the stale-image badge but never tells the user *how* to remediate it. Zero guidance on keeping workers up to date.
2. **The stale-badge tooltip** (`WorkersTab.tsx::ImageStaleBadge`, lines 125–151) points users at the Connect Worker modal — which emits a full `docker run` command valid for new installs but overkill for "I just need to refresh an already-running container". The cheap path (`docker pull && docker restart <name>`) isn't surfaced in the UI.

**Why not auto-update:** evaluated and declined. Docker has no truly native auto-update for long-running containers. Third-party tools (Watchtower is in maintenance limbo; What's Up Docker is active but still third-party) add friction and dependencies we don't want to own. Docker-native alternatives (Compose + scheduled pull; Swarm services + `docker service update --force`) all require user-configured schedulers anyway. For a home-lab tool the right trade-off is: don't promise automation we haven't built — document the manual path clearly, keep the stale-badge safety net, and let users wire their own pull cadence into whatever host scheduler they already use.

Explains why stale images matter (protocol compat, CVE patches, `MIN_IMAGE_VERSION` gating), how to spot them (Image stale badge + version cell), and ships three refresh recipes: `docker pull && docker restart` for plain Docker, `docker compose pull && up -d` for Compose, and a pointer to Connect Worker for full re-installs.
