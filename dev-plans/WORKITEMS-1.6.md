# Work Items — 1.6.0

Theme: **Power-user fleet management.** Git-backed config version history, per-device compile history, archived-config recovery, key/value device tagging, disk budgeting, and declarative worker routing — the controls you need to run a growing fleet without losing track of what it's doing.

## Settings

Introduce an in-app Settings surface separate from Home Assistant's Supervisor-managed `options.json`. Reasons:

- Feature toggles (auto-commit on save, job history retention, disk budgets, future proxy config) don't belong in the Supervisor Configuration tab — they're product settings, not deployment plumbing. Editing via Supervisor also forces a full add-on restart per change, which is hostile for day-to-day toggles.
- Over time most user-facing configuration should live here rather than in the Supervisor options form. `options.json` keeps only the fields Supervisor genuinely needs (token, port, `require_ha_auth`).

**Storage model.** `/data/settings.json`, owned by the server. Atomic write (tempfile + `os.replace`). Loaded once into an in-memory singleton; every mutation rewrites the file and updates the singleton under a single asyncio lock.

**Bootstrap.** On first boot after the 1.6 upgrade, if `/data/settings.json` doesn't exist, seed it once from the current `options.json` values for fields that have moved (`job_history_retention_days`, `firmware_cache_max_gb`, `job_log_retention_days`). After that the file is the only source of truth — editing the moved field in Supervisor has no effect. Documented in `DOCS.md`.

**Immediacy.** Every setting change takes effect immediately. Server-side consumers read `get_settings().xxx` at decision time (not at startup), so the next scheduled retention run, the next git auto-commit, the next cache prune all see the new value on the next tick — no add-on restart. UI consumers subscribe via an SWR hook on `/ui/api/settings`; the drawer's `mutate()` after a PATCH propagates changes to other tabs on the next poll.

## Auto-Versioning (local git, zero config)

Every save creates a git commit automatically. Users get per-file history, diff, and rollback without touching git or configuring anything. The config directory becomes a local git repo on first startup; existing git repos are left intact.

**Why git:** Diff is free (`git diff`), history is free (`git log --follow`), rollback is `git checkout <hash> -- <file>` + a new commit, Monaco already has a diff editor in the bundle, and the `git` binary ships in the HA add-on base image (zero image size cost). The alternative (copy-on-write snapshots + DIY diff) is more code and worse tooling.

- [x] **AV.1 Auto-init** *(1.6.0-dev.4, refined dev.5)* — on server startup, if `/config/esphome/` is not a git repo, run `git init` + `git add -A` + initial commit "Initial commit by distributed-esphome". If it's already a git repo (user set it up themselves), skip init entirely — including the `.gitignore`, which is left untouched so Pat-with-git's curated file isn't mutated. Fresh-init still writes `.gitignore` with `secrets.yaml` + `.esphome/` safety-net entries. Pre-existing repo with no resolvable `user.name`/`user.email` (nothing in repo/global/system config) gets a one-time Fleet identity fallback so auto-commits don't emit git's "configured automatically" warning.
- [x] **AV.2 Auto-commit on save** *(1.6.0-dev.4, refined dev.5, completed dev.10)* — editor save + **every other user-initiated write path** (pin / unpin, meta, schedule set / delete / toggle / once, rename, delete, archive restore) now triggers `git add --all -- <file>` + `git commit -m "<action>: <file>"` in a debounced async task. Rename commits both old and new paths so the history reads as a delete-of-old + add-of-new. Action label is specific per endpoint (e.g. `pin: bedroom.yaml`, `schedule: bedroom.yaml`) so `git log --author="HA User"` reads like an activity feed. **Gated on `settings.auto_commit_on_save`** (live-read at commit time). **Author respects the repo's own `user.name`/`user.email`**. Debounce window of 2s coalesces rapid actions. `.archive/` added to fresh-init gitignore so soft-deletions don't double-track. Non-blocking: each endpoint returns immediately, commits happen async.
- [x] **AV.3 History endpoint** *(1.6.0-dev.10)* — `GET /ui/api/files/{filename}/history` returns `[{hash, short_hash, date, author_name, author_email, message, lines_added, lines_removed}]` via `git log --follow --numstat`. The `--follow` flag tracks renames. Paginated via `limit` (default 50, max 500) + `offset`. Empty list (not an error) if the file has no commit history yet. Parser uses `\x1f` (unit separator) between fields — `\x1e` (record separator) breaks `str.splitlines()` in Python which treats it as a line boundary; the fix is tested.
- [x] **AV.4 Diff endpoint** *(1.6.0-dev.10)* — `GET /ui/api/files/{filename}/diff?from=<hash>&to=<hash>` returns `{"diff": "<unified diff>"}`. Works for any combination of hashes: both → direct diff; only `from` → diff against HEAD; only `to` → diff working tree against `to`; neither → diff working tree against HEAD. Hash values validated as 4–40 hex chars so injected shell metachars can't leak into `git diff` (tested).

**Build order (remaining).** AV.6 is the gateway that makes rollback and manual-commit actually usable — the endpoints for both (AV.5, AV.11) are built *with* the history panel so the Restore button and the "Commit these changes" affordance are live the moment the UI ships. AV.7 follows as its own pass. AV.8 folded into AV.6 — the panel IS the diff viewer. AV.9 stays deferred until the file-tree editor exists.

- [x] **AV.5 Rollback endpoint** *(1.6.0-dev.11)* — `POST /ui/api/files/{filename}/rollback` body `{hash}`. Runs `git checkout <hash> -- <filename>` to restore the working-tree file to that commit's content. **If `settings.auto_commit_on_save` is true**, follows up with `git add + git commit -m "revert: <file> to <short-hash>"` so the rollback is itself a recorded commit. When auto-commit is false, the working tree is left dirty — the user is expected to review + commit via AV.11. Returns `{content, committed, hash, short_hash}`. Invalidates the scanner config cache. Validates `hash` as 4-40 hex chars. Tests: 4 unit + 2 integration covering both toggle states, malformed hashes, and non-repo safety.

- [x] **AV.6 History + diff panel** *(1.6.0-dev.11, absorbs the old AV.8)* — the full-featured per-file history UX, opened from the Editor modal's toolbar ("History" button) and from the Devices-tab hamburger menu ("View history…"). One drawer/sheet scoped to one file:

  1. **Header row.** File name, commit count, and an auto-commit status chip (on/off + link to the Settings drawer). If there are uncommitted changes to this file, a yellow banner above the content reads *"You have uncommitted changes. [Commit…]"* and clicking the button opens AV.11's flow.

  2. **Compare selector.** A pill bar near the top: `From: [—] ↔ To: [Current]`. Both pills are dropdowns listing every commit plus "Current" (=working tree). Changing either side re-renders the diff below. Defaults: `From` = HEAD's parent (or the oldest available commit), `To` = Current — so opening the panel shows "what changed since the last committed state."

  3. **Diff viewer.** Monaco's built-in `DiffEditor` (already in the bundle, just not imported — this is what the retired AV.8 was going to do). Renders the unified diff between `From` and `To` via `GET /ui/api/files/{f}/diff?from=<hash>&to=<hash>`. Read-only, syntax-highlighted YAML, inline + side-by-side toggle. If the two sides are identical the editor shows a "no differences" empty state.

  4. **Commit list.** Newest-first table of commits (via AV.3), paginated with "Load more" at the bottom. Each row shows `short_hash · <action: filename> · <relative time> · +N/-M`. Shift/cmd-click a second row to set `From` to that commit instead. Per-row actions on hover/menu:
     - **Set as From / Set as To** — explicit compare
     - **Restore this version** — calls AV.5 with a confirmation dialog; shows what rollback will do (one-line summary: "Restore to {short_hash} from {relative time}. This will create a new `revert:` commit." or "…this will replace your uncommitted changes." when auto-commit is off).

  SWR-backed so a manual commit (AV.11) or a rollback refreshes the list without a full reload. Built on the shadcn `Sheet` primitive added in SP.4. Does NOT open inside the Editor modal (avoids a modal-in-modal situation); it's a separate drawer that can be opened with the Editor still underneath.

## Job History

Today the queue (`/data/queue.json`) IS the user's record of what the system did. Bug #18 removed time-based auto-pruning so history survives restarts, but per-target coalescing in `enqueue()` still clears *earlier* terminal jobs for a target whenever a new compile is enqueued (deliberate: one row per device on the Queue tab, not an ever-growing failure log). Net effect: users can only ever see the *latest* compile attempt per target, never the sequence of past attempts. Once they explicitly Clear the queue, even that disappears.

Add a persistent append-only job history that survives both coalescing AND explicit clears, powering: per-device "when did this last succeed?", fleet-wide success-rate rollups, compile-time trend lines, and audit trails of who-compiled-what-when. Separate from the live Queue (UI-wise and storage-wise), read-only from the user's perspective.

**Storage model:** SQLite (`/data/job-history.db`) rather than a growing JSON file. Per-job writes are cheap, indexed queries by target / timestamp / state are fast, and the file stays self-compacting. Add-on base image already has `sqlite3` as a stdlib builtin.

**Write model:** every time a job transitions to a terminal state (SUCCESS, FAILED, TIMED_OUT, CANCELLED), copy the finalized `Job` into the history DB. Coalescing still clears the queue entry but the history record stays. Explicit Clear on the Queue tab also leaves history intact.

- [ ] **JH.1 Schema + DAO** — new `ha-addon/server/job_history.py`. SQLite schema: `jobs(id TEXT PRIMARY KEY, target TEXT, state TEXT, triggered_by TEXT, trigger_detail TEXT, download_only BOOL, validate_only BOOL, pinned_version TEXT, esphome_version TEXT, assigned_client_id TEXT, submitted_at INTEGER, started_at INTEGER, finished_at INTEGER, duration_seconds REAL, compile_result TEXT, ota_result TEXT, config_hash TEXT, log_excerpt TEXT)`. `log_excerpt` is the last ~2 KB of the job log (not the full log — that's too big for an append-only DB). Indexed by `(target, finished_at DESC)` and `(finished_at DESC)`. DAO exposes `record_terminal(job)` + `query(target=None, state=None, since=None, limit=50, offset=0)`.
- [ ] **JH.2 Write hook** — `JobQueue.mark_result` / `timeout_checker` / `cancel` all funnel through a single `_transition_to_terminal(job, new_state)` helper today (verify); hook `job_history.record_terminal(job)` in that one place. Coalescing in `enqueue()` gets the same hook called right before the old entry is removed from `_jobs` (records the terminal state the removed job had at that moment). Idempotent on `id` — inserting the same job id twice is a no-op (ON CONFLICT DO NOTHING on the primary key).
- [ ] **JH.3 Retention** — `job_history_retention_days` lives in the new Settings store (see SP.*, default 365, max 3650, 0 = unlimited). Click a row → expands a log-excerpt panel. Paginates (load more) via `offset`. Empty state: "No history yet — first compile will appear here." ## Device Organization

Key/value tags (like AWS resource tags), stored in the per-device `# distributed-esphome:` comment block as a `tags:` map. Users can group the Devices table by any tag key (Notion-style table groups) and filter by `key=value`.

Format in the YAML comment block:
```yaml
# distributed-esphome:
#   tags:
#     location: kitchen
#     floor: "1"
#     env: prod
#     owner: stefan
```

The existing `tags` field landed in 1.4.0-dev.2 as a simple list of strings — that needs to migrate to a key/value map. `read_device_meta()` should accept both shapes during the transition (list → coerce to `{tag: ""}` or warn-and-ignore) and `write_device_meta()` always writes the map shape going forward.

## Disk Management

LRU-based disk usage controls for both the server and workers. Currently nothing caps the growth of ESPHome version caches, PlatformIO toolchains, compiled firmware, build directories, or job logs. On a worker with limited disk (e.g., a Raspberry Pi), these can silently fill the volume.

### Worker-side

- [ ] **DM.1 Worker disk budget** — new env var `MAX_DISK_USAGE_GB` (default: unlimited). On each job completion, the worker checks total usage of `/esphome-versions/` (versions + builds + PlatformIO). If over budget, evicts in LRU order: oldest unused ESPHome version venvs first (already has `MAX_ESPHOME_VERSIONS` for version count — this adds a size-based cap), then oldest build cache directories, then oldest PlatformIO packages. Logs what was evicted at INFO.
- [ ] **DM.2 Worker disk stats in heartbeat** — add `disk_total_mb`, `disk_used_mb`, `disk_free_mb` (for the `/esphome-versions` mount point) to the worker's `system_info` heartbeat. Server surfaces these on the Workers tab so operators can see when a worker is running low before it fails a compile.
- [ ] **DM.3 Build cache LRU** — the per-target build cache (`/esphome-versions/cache/<target>/`) currently grows unboundedly. Add LRU eviction: track last-access time per target cache dir, evict oldest when total build cache exceeds `MAX_BUILD_CACHE_GB` (env var, default 10GB). The existing `MAX_ESPHOME_VERSIONS` (count-based) stays for version venvs; this adds size-based eviction for the build artifacts.

### Server-side

- [ ] **DM.4 Server disk budget for caches** — `firmware_cache_max_gb` (default 2.0) and `job_log_retention_days` (default 30) live in the Settings store (see SP.*). Background task prunes `/data/firmware/` and old job logs on a daily schedule, reading current values from `get_settings()` each run (live-effect). Exposed in the Settings drawer under `Disk management`.
- [ ] **DM.5 Disk usage dashboard** — section on the Workers tab or a new Settings page showing: per-worker disk breakdown (versions, builds, PlatformIO, total), server-side cache sizes (PIO proxy cache, firmware, job logs), and the configured limits. Visual bar showing used/limit per category.

## Worker Constraints

Let users declare which workers can run which jobs. Originated in [issue #59](https://github.com/weirded/distributed-esphome/issues/59) ("Thread devices can't be reached from Windows desktop due to IPv6 limitation — can you auto-detect?"). **Reframed:** instead of trying to probe network reachability (fragile, slow, guesses wrong), let the user express their knowledge of the topology as declarative rules. This also generalizes to every other worker-selection need a user might have — "encrypted devices only go to the on-prem worker", "big configs only go to the beefy worker", "dev YAMLs only go to my laptop".

### Foundation: durable worker identity

The current `client_id` is an auto-generated UUID persisted to `/esphome-versions/.client_id` inside the worker's volume (`client.py:119,213-227`). If the volume wipes, the container gets rebuilt on a different host, or the user blows away their worker setup, they get a new UUID — breaking any saved config that referenced the old one. Worker constraints need a more durable identifier. The answer is: let the user name their workers.

- [ ] **WC.1 `WORKER_NAME` env var** — new optional env on the client (`client.py`). When set, it becomes the worker's primary identifier instead of the auto-UUID. When unset, fall back to the current auto-UUID behavior for backwards compatibility. `WORKER_NAME` values must match `^[a-z0-9][a-z0-9-]{0,63}$` (same slug rules as device names) so they're safe in URLs and UI chips.
- [ ] **WC.2 `WORKER_TAGS` env var** — comma-separated list of free-form tags, e.g. `WORKER_TAGS=ipv6,beefy,on-prem`. Sent at registration and on every heartbeat. Server surfaces them on `/ui/api/workers` for display and for constraint evaluation.
- [ ] **WC.3 Server: name-keyed registry** — `registry.py` accepts a `name` field on `RegisterRequest` (via `protocol.py` — **PROTOCOL_VERSION bump**, see note below). Registry key preference: `name` if provided, else `client_id`. If two workers register with the same `name`, the later one wins (logs a warning about the collision). Existing UUID-keyed workers continue to work unchanged.
- [ ] **WC.4 Protocol extension** — add `name: Optional[str]` and `tags: List[str]` to `RegisterRequest` and `HeartbeatRequest` in both `ha-addon/server/protocol.py` and `ha-addon/client/protocol.py` (byte-identical per PY-6). This is an **additive protocol change** — old workers sending neither field still register fine — so `PROTOCOL_VERSION` stays at its current value per the protocol.py docstring rule ("additive + optional unless PROTOCOL_VERSION is bumped").
- [ ] **WC.5 UI: show worker name + tags** — Workers tab's Hostname column becomes "Name / Hostname": shows `WORKER_NAME` prominently if set, falls back to hostname. New toggleable "Tags" column rendering tags as chips. Connect Worker modal's generated `docker run` command includes `-e WORKER_NAME=<slug>` and an optional `-e WORKER_TAGS=<tags>` pre-filled with a hint.

### Constraint expression

Declarative matching between targets and workers, stored in the per-device `# distributed-esphome:` YAML comment block (same pattern as tags, pin_version, schedules). Constraints are **additive to the existing `pinned_client_id`** — pinning a specific worker on a job still wins over general constraints.

- [ ] **WC.6 Target-side constraint fields** — extend the per-device metadata comment:
  ```yaml
  # distributed-esphome:
  #   worker_requires:       # worker must have ALL of these tags
  #     - ipv6
  #   worker_forbids:        # worker must have NONE of these tags
  #     - cloud
  #   worker_only:           # whitelist by worker name (overrides tags if set)
  #     - home-beefy
  ```
  `read_device_meta` / `write_device_meta` extended to parse + emit these three fields. Surfaced on `/ui/api/targets` as `worker_requires`, `worker_forbids`, `worker_only`.

### Evaluation

- [ ] **WC.7 `claim_next` constraint filter** — in `job_queue.claim_next()`, before a worker can claim a job, check whether its name/tags satisfy the job's target constraints. If not, the job is skipped for that worker and remains PENDING for the next eligible worker to claim. The existing `pinned_client_id` check runs first (explicit pin wins); constraints run second.
- [ ] **WC.8 "No eligible worker" detection** — when a job has been PENDING for > N minutes AND no currently-online worker satisfies its constraints, mark it FAILED with a clear message: *"No eligible worker: target requires tags [ipv6]; online workers: home-pi (tags: []), office-desktop (tags: [beefy])"*. This is the "stuck forever" failure mode — better to fail loudly than hang.

### Notes & non-goals

- **Not auto-detection.** This is deliberately user-declarative. We don't probe reachability, we don't ping from each worker, we don't parse YAML for `manual_ip` to infer things. Users know their network better than we do, and probe-based logic is a maintenance tax.
- **Backwards compatibility.** Workers without `WORKER_NAME` keep UUIDs. Targets without `worker_requires`/`worker_forbids`/`worker_only` accept any worker. Zero breaking changes.
- **Relationship to `pinned_client_id`.** Explicit pin in the UpgradeModal still overrides everything — that's a one-shot manual choice and shouldn't be filtered out by general constraints. Constraints are the persistent default; pins are the override.
- **Related, out of scope:** per-worker `WORKER_CAN_RUN` whitelists (the inverse: worker declares which targets it accepts). Makes sense for shared workers in mixed-trust environments but not needed for the home-lab use case. Revisit in a future release if requested.

## Worker Update Documentation

After the SC.4 revert (bug #58 in 1.4.1), workers update manually: user notices the stale badge in the Workers tab and runs `docker pull && docker restart` on the worker host. Two gaps today:

1. **`ha-addon/DOCS.md`** (`line 21`) mentions the stale-image badge but never tells the user *how* to remediate it. Zero guidance on keeping workers up to date.
2. **The stale-badge tooltip** (`WorkersTab.tsx::ImageStaleBadge`, lines 125–151) points users at the Connect Worker modal — which emits a full `docker run` command valid for new installs but overkill for "I just need to refresh an already-running container". The cheap path (`docker pull && docker restart <name>`) isn't surfaced in the UI.

**Why not auto-update:** evaluated and declined. Docker has no truly native auto-update for long-running containers. Third-party tools (Watchtower is in maintenance limbo; What's Up Docker is active but still third-party) add friction and dependencies we don't want to own. Docker-native alternatives (Compose + scheduled pull; Swarm services + `docker service update --force`) all require user-configured schedulers anyway. For a home-lab tool the right trade-off is: don't promise automation we haven't built — document the manual path clearly, keep the stale-badge safety net, and let users wire their own pull cadence into whatever host scheduler they already use.

Covers: why it matters (protocol compatibility, CVE patches, `MIN_IMAGE_VERSION` gating), how to tell (stale badge + per-worker build version), refresh command (`docker pull ghcr.io/weirded/esphome-dist-client:latest && docker restart <name>`) with Windows/Mac Docker Desktop parity note, full re-install pointer to Connect Worker modal, Docker Compose variant (`docker compose pull && docker compose up -d`), one-paragraph aspirational pointer at community auto-update tools (Watchtower/wud/cron) with explicit "we don't endorse a specific one", optional cosign verification link pointing at SECURITY.md.
- [ ] **WU.2 Improve stale-badge tooltip** — `ha-addon/ui/src/components/WorkersTab.tsx:125-151`.
